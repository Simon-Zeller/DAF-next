# Feature Specification: AI Semantic Layer Crew

**Feature Branch**: `009-ai-semantic-layer-crew`
**Created**: 2026-03-12
**Status**: Draft
**Input**: User description: "Feature: AI Semantic Layer Crew. The eighth CrewAI crew
in the DAF pipeline (Phase 5 â€” runs after Governance Crew; has no strict ordering
dependency relative to the Analytics Crew, so both Phase 5 crews may run in either
order). Purpose: expose the design system as a machine-readable knowledge base for AI
code generators and coding assistants. All outputs are static JSON or config files â€”
no source code, no token files, no doc pages are produced."

## User Scenarios & Testing *(mandatory)*

### User Story 1 â€” Component Registry Enables AI-Assisted Component Usage (Priority: P1)

A developer opens their code editor and starts typing a new component. Their AI
coding assistant (Cursor, GitHub Copilot, or another LLM-backed tool) suggests the
correct props, variants, and usage pattern for `Button` without the developer needing
to read the design system documentation separately. The assistant knows the component
API, the valid token bindings for each prop, the required a11y attributes, and the
available states. This is powered by `registry/components.json`, which the AI context
files reference as their primary data source.

**Why this priority**: The component registry is the foundation of the entire semantic
layer. Every other agent's output (`registry/tokens.json`, `.cursorrules`,
`copilot-instructions.md`, `ai-context.json`) depends on or references it. Without an
accurate, comprehensive registry, AI assistants will hallucinate props, suggest
deprecated patterns, or produce non-compliant component trees. It is the highest-value
output on the critical path.

**Independent Test**: Run Agent 41 alone against a pipeline output for a single
component (e.g., `Button`). Verify `registry/components.json` exists and contains a
top-level `Button` entry with: a `props` array covering all props from
`Button.spec.yaml`, a `variants` array matching all variant values from the spec, a
`states` array, a `tokenBindings` object with at least one entry per prop that uses
a design token, an `a11yAttributes` array, and at least one `usageExample` string.

**Acceptance Scenarios**:

1. **Given** a pipeline output with 26 generated components, **When** Agent 41 builds
   the component registry, **Then** `registry/components.json` contains an entry for
   every component in `reports/generation-summary.json`, with zero components omitted.
2. **Given** a component spec YAML that defines a `size` prop with values
   `sm | md | lg` and a default of `md`, **When** Agent 41 processes that component,
   **Then** the registry entry for that component contains a `props` entry for `size`
   with `type: "sm | md | lg"`, `default: "md"`, and a plain-language `description`
   field.
3. **Given** a component spec YAML that defines token bindings (e.g., `background`
   mapped to `color.surface.primary`), **When** Agent 41 processes that component,
   **Then** the `tokenBindings` object in the registry entry maps each prop to its
   resolved token name â€” not a raw hex value.
4. **Given** a component spec YAML that documents a11y requirements (e.g., `role`,
   `aria-label` usage), **When** Agent 41 processes it, **Then** the registry entry
   has an `a11yAttributes` array containing each documented attribute with a
   description of when it is required.
5. **Given** a component that has no spec YAML (it exists in source but was not
   generated by the pipeline), **When** Agent 41 scans the output, **Then** that
   component is skipped â€” the registry covers only spec-defined components â€” and a
   warning entry is added to the crew's run log.
6. **Given** Agent 41 completes successfully, **When** `registry/components.json` is
   parsed, **Then** it is valid JSON, and a top-level `generatedAt` timestamp field
   and a `componentCount` field matching the number of registry entries are both
   present.

### User Story 2 â€” Token Resolution Graph Enables Intent-to-Token Lookups (Priority: P1)

A developer asks their AI assistant: "What token should I use for a muted background
color?" The assistant consults `registry/tokens.json` and responds: "Use
`color.surface.muted` â€” resolved value `#F5F5F5` on light theme, `#1A1A1A` on dark
theme. This is a semantic tier token in the surface color category, related to
`color.surface.primary` and `color.surface.overlay`." The developer gets an accurate,
actionable answer without leaving their editor.

**Why this priority**: Token resolution is the second critical foundation. AI
assistants without token graph access default to suggesting hardcoded hex values or
vague variable names. The token graph transforms semantic intent into the correct
token reference on the first suggestion, preventing the compliance violations the
Analytics Crew would otherwise flag.

**Independent Test**: Run Agent 42 alone against a pipeline output. Verify
`registry/tokens.json` exists and contains at least one entry for each token in
`tokens/semantic.tokens.json`. For one token, assert: `resolvedValue` includes at
least a `default` platform entry, `tier` is one of `global|semantic|component`,
`semanticPurpose` is a non-empty string, `relatedTokens` is an array (can be empty),
and `naturalLanguageDescription` is a non-empty human-readable sentence.

**Acceptance Scenarios**:

1. **Given** `tokens/semantic.tokens.json` contains a token named
   `color.text.secondary`, **When** Agent 42 builds the token graph, **Then**
   `registry/tokens.json` contains an entry for `color.text.secondary` with a
   `naturalLanguageDescription` like "Secondary text color, used for supporting
   content and captions."
2. **Given** a token exists in three tier files (global, semantic, component-scoped),
   **When** Agent 42 processes the token set, **Then** the graph covers tokens from
   all three tier files â€” zero tokens defined in any tier file are absent from the
   graph.
3. **Given** a semantic token that maps to a global primitive (e.g.,
   `color.surface.primary â†’ color.blue.50`), **When** Agent 42 resolves the token
   chain, **Then** the `resolvedValue` field contains the fully resolved value (the
   primitive's actual value), not the intermediate alias.
4. **Given** a token with a `deprecated: true` flag in its tier file, **When** Agent
   42 processes it, **Then** the graph entry includes a `deprecated: true` field and
   a `replacementToken` field naming the recommended replacement.
5. **Given** two tokens in the same semantic category (e.g., `color.surface.primary`
   and `color.surface.muted`), **When** Agent 42 builds the graph, **Then** each
   entry's `relatedTokens` array includes the other token in the same category.

### User Story 3 â€” Composition Rules Prevent Invalid Component Trees (Priority: P2)

A developer asks their AI assistant to compose a form layout. The assistant generates
a `Form` wrapping a `Select` wrapping a `Button`. The coding assistant catches this
before writing the code: `Button` is not a valid child of `Select` according to
`registry/composition-rules.json`. It instead suggests wrapping the `Button` in a
`FormActions` slot. The tree arrives in the codebase structurally valid on the first
generation.

**Why this priority**: Composition constraints are P2 because they prevent a class of
structural errors that are invisible to static type checkers but visible to design
reviewers and accessibility auditors. Without this registry, AI assistants generate
technically valid TypeScript but semantically invalid component trees. It is P2
because its value compounds over time (as the component library grows) rather than
delivering immediate correctness for every component on day one.

**Independent Test**: Run Agent 43 alone. Verify `registry/composition-rules.json`
exists and contains at least one component entry with: an `allowedChildren` array,
a `forbiddenNesting` array, a `requiredSlots` array, a `maxDepth` number, and at
least one `validExample` and one `invalidExample` composition tree. Assert both
example trees are valid JSON structures.

**Acceptance Scenarios**:

1. **Given** a component spec YAML that specifies allowed children (e.g.,
   `TabPanel` accepts `TabItem` children only), **When** Agent 43 processes it,
   **Then** `registry/composition-rules.json` contains a `TabPanel` entry with
   `allowedChildren: ["TabItem"]`.
2. **Given** a component that has no explicit composition constraints in its spec,
   **When** Agent 43 processes it, **Then** an entry is still created with
   `allowedChildren: []` (unrestricted), `forbiddenNesting: []`, `requiredSlots: []`,
   and `maxDepth: null` â€” absence of constraint is explicit, not absent.
3. **Given** a component with a required named slot (e.g., `Modal` requires a
   `title` slot), **When** Agent 43 processes it, **Then** the `requiredSlots` array
   contains `"title"` and the `validExample` tree includes a node occupying the
   `title` slot.
4. **Given** `registry/composition-rules.json` is generated, **When** it is parsed as
   JSON, **Then** all `validExample` and `invalidExample` composition trees are
   themselves valid JSON objects â€” no embedded string-escaped JSON trees.

### User Story 4 â€” Compliance Rules Enable Linter and AI Validation (Priority: P2)

A team adopts the generated design system. They configure their CI pipeline to use
`registry/compliance-rules.json` as a rule source for their custom linter. Any pull
request that introduces a hardcoded hex value, a forbidden component nesting, a
missing aria attribute, or a non-standard naming convention fails CI automatically.
The rules need no manual authoring â€” they are derived directly from the design
system's own specs and governance output.

**Why this priority**: Compliance rules are the enforcement mechanism that gives the
generated design system long-term structural integrity. Without them, the design
system rules exist only in documentation that developers can ignore. Exporting them
as a structured, machine-readable file makes the rules enforceable in any toolchain
without requiring the team to author the rules by hand.

**Independent Test**: Run Agent 44 alone against a full pipeline output. Verify
`registry/compliance-rules.json` exists and contains at least one rule entry in each
of the four categories: `tokenUsage`, `compositionRules`, `a11yRequirements`,
`namingConventions`. Assert each rule entry has a unique `ruleId`, a `severity`
field (`error | warning | info`), a `description` field, and a `rationale` field.

**Acceptance Scenarios**:

1. **Given** the token compliance rules from the Analytics Crew are available in
   `reports/token-compliance.json`, **When** Agent 44 compiles the compliance rules,
   **Then** `registry/compliance-rules.json` contains `tokenUsage` rules that prohibit
   the hardcoded value types (hex, raw px, deprecated token refs) identified in the
   analytics report â€” the rules are derived from the system, not hardcoded.
2. **Given** composition constraints exist in `registry/composition-rules.json`,
   **When** Agent 44 compiles compliance rules, **Then** each `forbiddenNesting` pair
   has a corresponding `compositionRules` entry in `registry/compliance-rules.json`
   with `severity: error`.
3. **Given** a component's spec YAML documents required a11y attributes, **When**
   Agent 44 compiles rules for that component, **Then** a `a11yRequirements` rule is
   generated for each required attribute with a `severity: error` and a description
   explaining when the attribute is required.
4. **Given** `registry/compliance-rules.json` is generated, **When** all rule entries
   are inspected, **Then** every `ruleId` is unique across all four categories â€” no
   duplicate rule IDs exist.

### User Story 5 â€” AI Context Files Drop Into Cursor and Copilot Without Manual Authoring (Priority: P3)

After the pipeline completes, a developer opens the output folder, copies `.cursorrules`
to their project root and `copilot-instructions.md` to `.github/`, and copies
`registry/` to a shared location. Without any manual editing, their AI coding
assistant sessions now suggest components using the correct prop names, the correct
token references, and the correct composition patterns. The context was optimized for
the token budgets of the target assistants â€” the Cursor file uses Cursor's syntax;
the Copilot file is structured for GitHub Copilot's instruction format.

**Why this priority**: Context serialization is P3 because it is the packaging step
that depends on all prior agents completing successfully. Its output is the most
user-visible artifact, but it cannot be tested or validated in isolation. If the
registry, token graph, and rules are correct, the serialized files will be correct by
construction.

**Independent Test**: Run Agent 45 after Agents 41â€“44 have produced their registry
files. Verify `.cursorrules`, `copilot-instructions.md`, and `ai-context.json` all
exist and are non-empty. Assert `.cursorrules` contains at least one component name
from the registry. Assert `copilot-instructions.md` opens with a valid Markdown
heading. Assert `ai-context.json` is valid JSON containing a top-level `components`
key and a top-level `tokens` key.

**Acceptance Scenarios**:

1. **Given** all four registry files exist (`registry/components.json`,
   `registry/tokens.json`, `registry/composition-rules.json`,
   `registry/compliance-rules.json`), **When** Agent 45 serializes them, **Then**
   `.cursorrules`, `copilot-instructions.md`, and `ai-context.json` are all written
   to the output root.
2. **Given** the full component registry with 26 components is large, **When** Agent
   45 applies token budget optimization, **Then** `.cursorrules` and
   `copilot-instructions.md` contain a concise representation â€” either summarizing
   component APIs or grouping by category â€” rather than a byte-for-byte copy of the
   raw registry JSON; `ai-context.json` may include the full data.
3. **Given** `.cursorrules` is generated, **When** it is opened, **Then** it follows
   Cursor's documented rule file format (starts with a rules block, uses the expected
   syntax for rule definitions), so that Cursor parses it without errors.
4. **Given** `copilot-instructions.md` is generated, **When** it is opened in a
   Markdown reader, **Then** it renders without errors and contains clearly identified
   sections for: component usage rules, token usage rules, composition constraints,
   and naming conventions.
5. **Given** `ai-context.json` is generated, **When** it is parsed, **Then** it is
   valid JSON with a top-level `generatedAt` timestamp, a `components` array, a
   `tokens` array, a `compositionRules` object, and a `complianceRules` array.

### Edge Cases

- `reports/composition-audit.json` is absent â€” Agent 43 derives composition rules
  from spec YAMLs only (the `composition` section in each spec, if present); the
  `registry/composition-rules.json` is still written, with a note in the crew run log
  that the composition audit was unavailable and some constraint entries may be
  incomplete.
- A spec YAML has no `props` section (e.g., a purely visual wrapper component with
  no configurable props) â€” Agent 41 writes a registry entry with an empty `props`
  array; this is valid and not treated as an error.
- A token tier file is malformed JSON â€” Agent 42 skips that tier file, records a
  `source-error` entry in the crew run log, and continues with all other tier files.
  Tokens from the malformed file are absent from `registry/tokens.json`; the file is
  still written with all successfully parsed tokens.
- Agent 45 runs but `registry/composition-rules.json` is absent (Agent 43 failed) â€”
  Agent 45 serializes the available registry files (components, tokens, compliance
  rules) and marks the composition section as `unavailable` in all three context
  output files rather than failing the entire serialization run.
- A component name in the registry contains characters unsafe for the `.cursorrules`
  format (e.g., a component named `3DIcon` starting with a digit) â€” Agent 45 applies
  a safe identifier transformation (e.g., prefix `C_3DIcon`) and records the mapping
  in the crew run log.
- `registry/components.json` contains zero components (Agent 41 found no valid spec
  YAMLs) â€” Agent 45 fails fast: it does not write `.cursorrules`,
  `copilot-instructions.md`, or `ai-context.json` with empty data. Instead, it writes
  an error marker file `registry/.error` explaining the failure, and the crew run
  record is marked as `failed`.

## Requirements *(mandatory)*

### Functional Requirements

**Agent 41 â€” Registry Maintenance Agent**

- **FR-001**: Agent 41 MUST write `registry/components.json` containing one entry per
  spec-defined component (components that exist in `specs/*.spec.yaml` and have a
  corresponding TSX source file in `src/components/`).
- **FR-002**: Each registry entry MUST include: component name, `props` array (each
  prop with `name`, `type`, `default`, `required` boolean, `description`), `variants`
  array, `states` array, `slots` array, `tokenBindings` object (prop name â†’
  resolved token name), `a11yAttributes` array (each with `name`, `when` condition,
  `description`), and `usageExamples` array (at least one example per component).
- **FR-003**: Token bindings in the registry MUST reference resolved token names
  (e.g., `color.surface.primary`), not raw values. Agent 41 resolves the binding
  reference via the token tier files.
- **FR-004**: `registry/components.json` MUST include a top-level `generatedAt`
  ISO-8601 timestamp and a `componentCount` integer matching the number of entries.
- **FR-005**: Agent 41 MUST NOT write to any path outside of `registry/`.

**Agent 42 â€” Token Resolution Agent**

- **FR-006**: Agent 42 MUST write `registry/tokens.json` covering every token defined
  in all three tier files (`tokens/base.tokens.json`, `tokens/semantic.tokens.json`,
  `tokens/component.tokens.json`).
- **FR-007**: Each token entry MUST include: `name`, `tier` (`global|semantic|component`),
  `resolvedValue` object (at minimum a `default` key with the token's fully resolved
  value), `semanticPurpose` (non-empty descriptive string), `relatedTokens` array
  (tokens in the same category), `naturalLanguageDescription` (complete sentence
  suitable for an AI context window), and a `deprecated` boolean.
- **FR-008**: Deprecated tokens MUST include a `replacementToken` field. If no
  replacement is defined, the value is `null` and a warning is added to the run log.
- **FR-009**: `resolvedValue` MUST represent the fully resolved leaf value â€” not an
  alias reference. Agent 42 follows the full alias chain to the primitive value.
- **FR-010**: Agent 42 MUST NOT write to any path outside of `registry/`.

**Agent 43 â€” Composition Constraint Agent**

- **FR-011**: Agent 43 MUST write `registry/composition-rules.json` with one entry per
  spec-defined component. Components with no composition constraints receive an entry
  with `allowedChildren: []`, `forbiddenNesting: []`, `requiredSlots: []`,
  `maxDepth: null`.
- **FR-012**: Each entry MUST include at least one `validExample` and one
  `invalidExample` composition tree (for components that have constraints). Components
  with no constraints may omit these fields or provide them as null.
- **FR-013**: Composition rules MUST be derived from the spec YAML `composition`
  section and, when available, from `reports/composition-audit.json`. Spec is
  authoritative â€” composition audit provides supplementary evidence only.
- **FR-014**: Agent 43 MUST NOT write to any path outside of `registry/`.

**Agent 44 â€” Validation Rule Agent**

- **FR-015**: Agent 44 MUST write `registry/compliance-rules.json` containing rules
  in four categories: `tokenUsage`, `compositionRules`, `a11yRequirements`,
  `namingConventions`.
- **FR-016**: Each rule entry MUST include: `ruleId` (unique across all categories),
  `category`, `severity` (`error|warning|info`), `description`, `rationale`, and
  optionally `autoFixable` boolean.
- **FR-017**: All `ruleId` values MUST be unique across the entire file â€” no duplicates
  across categories.
- **FR-018**: `tokenUsage` rules MUST cover at minimum: prohibition of hardcoded hex
  colors, prohibition of hardcoded spacing values, prohibition of deprecated token
  references.
- **FR-019**: Agent 44 MUST NOT write to any path outside of `registry/`.

**Agent 45 â€” Context Serializer Agent**

- **FR-020**: Agent 45 MUST write three output files: `.cursorrules` (Cursor rules
  format), `copilot-instructions.md` (GitHub Copilot instructions format), and
  `ai-context.json` (generic unified format).
- **FR-021**: `.cursorrules` MUST follow Cursor's documented rule file format. It MUST
  include component usage rules, token reference rules, and composition constraints.
- **FR-022**: `copilot-instructions.md` MUST be valid Markdown with a top-level
  heading and clearly separated sections for: component usage, token usage,
  composition rules, and naming conventions.
- **FR-023**: `ai-context.json` MUST be valid JSON with top-level keys: `generatedAt`,
  `components`, `tokens`, `compositionRules`, `complianceRules`. Missing source
  sections (e.g., if Agent 43 failed) MUST be represented as `null` with an
  `errors` array at the top level noting what is missing.
- **FR-024**: Agent 45 MUST apply token budget optimization for `.cursorrules` and
  `copilot-instructions.md` â€” these files MUST NOT be a byte-for-byte copy of the
  raw registry JSON. Optimization strategies include: summarizing prop type unions,
  grouping components by category, omitting zero-constraint composition entries.
  `ai-context.json` may include the full unoptimized data.
- **FR-025**: If `registry/components.json` is absent or empty, Agent 45 MUST NOT
  write `.cursorrules`, `copilot-instructions.md`, or `ai-context.json`. It MUST
  write `registry/.error` with an explanation and mark the crew run as `failed`.
- **FR-026**: Agent 45 output files MUST be written to the output root (`./`) for
  `.cursorrules` and `copilot-instructions.md`, and to `registry/` for the four
  registry JSON files and `ai-context.json`. No writes to `src/`, `specs/`,
  `tokens/`, `docs/`, or `governance/`.

**Crew-Level**

- **FR-027**: Tasks execute in the order T1 â†’ T2 â†’ T3 â†’ T4 â†’ T5. Agents 41â€“44 may
  run in parallel (T1â€“T4 have no mutual dependency), but Agent 45 MUST wait for all
  four registry files.
- **FR-028**: If any of Agents 41â€“44 fail, Agent 45 serializes whichever registry
  files are available and marks the unavailable sections as `null` in `ai-context.json`
  â€” it does not skip serialization entirely unless `registry/components.json` is
  absent (see FR-025).
- **FR-029**: The crew MUST append a crew-level run record to
  `reports/generation-summary.json` on completion, recording: start time, end time,
  per-agent status (pass/fail), and count summaries (components in registry, tokens
  in graph, rules generated, output files written).

### Key Entities

- **Component Registry** (`registry/components.json`): The machine-readable API
  surface of the entire design system. One entry per spec-defined component, covering
  the full prop API, variants, states, slots, token bindings, a11y requirements, and
  usage examples. The primary input to Agent 45.
- **Token Graph** (`registry/tokens.json`): A traversable graph of all tokens in
  the design system, enriched with semantic purpose descriptions and natural language
  summaries suitable for AI context windows.
- **Composition Rules** (`registry/composition-rules.json`): Machine-readable
  constraints on valid component trees. Records what may nest inside what, what must
  not nest inside what, and what slots are required.
- **Compliance Rules** (`registry/compliance-rules.json`): Exportable, linter-ready
  rules derived from the design system's own spec and governance output. Structured
  for consumption by automated tooling without manual rule authoring.
- **AI Context Files**: The three consumer-facing output files â€” `.cursorrules` for
  Cursor, `copilot-instructions.md` for GitHub Copilot, and `ai-context.json` for
  generic LLM tools. These are the artifacts developers drop into their projects to
  activate design-system-aware AI assistance.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `registry/components.json` covers 100% of components in
  `reports/generation-summary.json` â€” zero spec-defined, successfully generated
  components are absent from the registry.
- **SC-002**: `registry/tokens.json` covers 100% of tokens defined across all three
  tier files â€” zero defined tokens are absent from the graph.
- **SC-003**: Every token entry in `registry/tokens.json` has a non-empty
  `naturalLanguageDescription` â€” zero entries with blank or null description fields.
- **SC-004**: All compliance rule IDs in `registry/compliance-rules.json` are unique
  â€” zero duplicate rule IDs across all four categories.
- **SC-005**: `.cursorrules`, `copilot-instructions.md`, and `ai-context.json` are all
  valid (parseable and non-empty) after every successful crew run.
- **SC-006**: Agent 45 applies token budget optimization: `.cursorrules` and
  `copilot-instructions.md` are each smaller than the combined byte size of the four
  raw registry JSON files â€” demonstrating that optimization was applied rather than
  verbatim copying.

## Assumptions

- Agent 41 uses the spec YAML as the authoritative source for the component API. TSX
  source is read for supplementary details (e.g., inferring type shapes from prop
  types), but the spec is authoritative for what goes into the registry.
- Agent 42 resolves token aliases by following the full chain from semantic/component
  tier through global (primitive) tier until a leaf value is reached. Circular alias
  chains are recorded as `resolution-error` entries without crashing the agent.
- Agent 43 derives composition constraints primarily from the `composition` section in
  each spec YAML. If the spec has no composition section, the component is treated as
  having no constraints (valid everywhere, unrestricted children).
- Agent 45's token budget optimization is heuristic â€” it summarizes and groups, but
  does not guarantee a specific file size. The goal is human-readable conciseness, not
  a hard character limit.
- The AI Semantic Layer Crew has no strict ordering dependency relative to the
  Analytics Crew. Both are Phase 5 crews and may run in either order. The AI Semantic
  Layer Crew does not read from or depend on the Analytics Crew's outputs
  (`reports/token-usage.json`, `reports/token-compliance.json`,
  `reports/drift-report.json`, `reports/pipeline-completeness.json`).
- `.cursorrules` format is Cursor's documented rule file format as of the pipeline
  implementation date. If Cursor changes its format, Agent 45 requires a targeted
  update.

## Out of Scope

- Generating or modifying component source code â€” this crew reads `src/` but never
  writes to it.
- Generating or modifying token files â€” this crew reads `tokens/` but never writes
  to it.
- Generating or modifying doc pages â€” documentation is the Documentation Crew's
  responsibility.
- Hosting or serving the registry as an API â€” all outputs are static files on disk.
- Validating generated component code against the compliance rules â€” that is the
  Analytics Crew's and Governance Crew's responsibility; this crew only exports the
  rules in machine-readable form.
- Runtime AI assistant integration or plugin development â€” the crew produces static
  context files; integration into a specific IDE or assistant is the consumer's
  responsibility.
